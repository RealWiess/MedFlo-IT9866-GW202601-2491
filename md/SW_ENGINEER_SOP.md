# MedFlow 軟體工程師開發規範與核心技能 (Software Engineer SOP)

為了確保專案的長期維護性、程式碼品質以及團隊開發的順暢，所有參與本目錄 (`C:\SW code\source code`) 下專案開發的軟體工程師 (包含 AI 助理) 都必須具備並嚴格遵守以下核心技能與規範：

## 1. 總工程師的架構思維 (Chief Software Engineer Mindset)
* **主動提供架構級建議**：面對新需求或除錯時，不能只是盲目執行命令或採用「硬幹 (Hardcode)」的解法。必須提出符合業界標準、具備擴展性與穩定性的「正確作法」或「架構方案」。
* **決策前置評估**：在動手大幅修改底層架構前，必須先將專業提案交由主導者評估與決策。
* **保持專業與前瞻性**：隨時關注程式碼的品質、系統資源的使用率（如避免無窮迴圈阻塞匯流排）、除錯容易度，以及未來維護的成本。

## 2. 版本控制與備份管理 (Version Control & Build Backup)
* **單一目錄與 Git 管理**：不再使用「複製資料夾加上日期」的土法煉鋼方式備份。所有的開發都應該在固定的專案目錄下進行。
* **嚴格的 Commit 紀律**：每天完成修改後，必須使用 `git add .` 和 `git commit` 來提交程式碼，清楚寫明修改原因。
* **版本標記**：可以使用 `git tag <日期>`（例如 `20260708`）來標記當天穩定版本；若需要進行破壞性實驗，必須開立新的 Git Branch 進行。
* **ROM 檔追溯機制**：韌體編譯完成後，產出的 `.ROM` 檔案必須複製到專案資料夾根目錄，並且重新命名為帶有時間戳記的格式（例如 `GW202601_YYYYMMDD_HHMMSS.ROM`），以供未來產線燒錄或問題追溯。

## 3. 研發日誌維護 (Development Logging)
* **必備文件**：每一個案子在其專案資料夾根目錄都必須建立一個研發日誌（例如 `DEVELOPMENT_LOG.md`）。
* **記錄內容**：每次進行重大除錯、修改架構或完成特定任務後，必須將「修改的內容」、「遇到的 Bug 根本原因」、以及「採取該解決方案的理由」詳細記錄下來。
* **避免重複踩坑**：這份日誌是團隊的共同資產，能夠有效避免未來開發者忘記先前的決策而犯下同樣的錯誤。

## 4. 修改前的「三步驟查閱」規範 (Pre-Modification Review)
在進行任何專案的架構修改或新功能開發前，工程師必須優先執行以下三步驟查閱，絕不可憑空想像或覆寫已正確運作的邏輯：
1. **閱讀研發日誌 (`DEVELOPMENT_LOG.md`)**：掌握前人踩過的雷與除錯血淚史。
2. **閱讀系統規格書 (`GATEWAY_SPEC.md` 等)**：了解目前的規格、通訊協定與過濾條件等，確保新的修改不會破壞既有規格。
3. **查閱 Git 歷史紀錄**：使用 `git log` 或 Github 介面回顧先前的 Commit 歷史，確認之前的修改歷程與決策原因，找出已經驗證為正確的版本邏輯。
